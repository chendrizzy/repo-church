# Market Landscape and Watchlist

## Landscape Summary

Agentic coding tools increasingly need durable project context, gateable plans, evidence capture, and handoffs that survive across model sessions and runtimes. Repo Church positions itself as a portable lifecycle and artifact package rather than a hosted runtime: it gives agents deterministic state, ledgers, skill contracts, and a Repo Bible that can be installed into compatible agent environments.

## Threat and Opportunity Matrix

| Area | Threat | Opportunity | Evidence | Watch cadence |
|---|---|---|---|---|
| Generic planning prompts | Agents can produce plausible but shallow plans without proof. | Make proof, owner, and recheck fields non-optional in gates and profiles. | `skills/church/references/gate-taxonomy.md`, `commands/*.md` | Each release |
| Runtime lock-in | Vendor-specific workflows can trap repo process in one agent. | Keep skills, commands, agents, and CLI portable. | `AGENTS.md`, `.claude-plugin/plugin.json`, `skills/church/scripts/church.py` | Each package change |
| Context bloat | Large doctrine packets can overwhelm model context. | Use CLI inventory, sources, claim scan, HTML render, and compact context output. | `skills/repo-bible/scripts/repo_bible.py`, `_repo-bible-inventory.md` | Each Bible refresh |
| Weak moat claims | Strategy language can overclaim without proof. | Require local evidence and validation tests in moat scoring. | `.church/moat/moat.json`, `score_moat` | Each moat update |

## Gaps and Biggest Opportunities

- External competitor research is not yet current-source-backed in this meta run; keep market claims framed as category-level and local-evidence-backed.
- Strongest near-term opportunity: turn self-hosted lifecycle proof into a public trust signal for agent workflow packages.
- Biggest risk: the framework becomes ceremonial if every gate can be satisfied by narrative instead of command/test evidence.
