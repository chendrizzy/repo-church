<purpose>
This reference defines the research standard for market-backed Bible generation.
</purpose>

<when_to_browse>
Browse current sources when facts can drift:

- Competitor features, pricing, docs, customers, integrations, policies, security posture.
- Market share, ecosystem counts, funding, launch status, partner programs.
- Laws, standards, protocols, platform review rules, app store policies.
- Buyer behavior, benchmarks, usage metrics, and trend claims.
</when_to_browse>

<source_priority>
Prefer sources in this order:

1. Official docs, pricing pages, changelogs, API references, specs.
2. Peer-reviewed papers, credible benchmarks, security advisories.
3. Company blogs or press releases for self-claims, clearly labeled.
4. Investor/analyst reports when public and dated.
5. High-quality secondary coverage for context, not as sole proof.
</source_priority>

<research_outputs>
For every important competitor/threat, capture:

- Product/category.
- Current capabilities.
- Buyer/user.
- Pricing/economic model.
- Distribution channels.
- Where it overlaps the project.
- Where it commoditizes the project.
- Where the project can still win.
- Source URLs and access date.
- Trigger thresholds for future watch.
</research_outputs>

<historical_success_pattern>
When borrowing from historical success stories:

- Extract principles, not surface tactics.
- Verify the story with current or credible sources.
- Check whether the era/context still applies.
- Translate the principle into a project-specific gate.
- Avoid copying a strategy when buyer, channel, margin, or timing differs.
</historical_success_pattern>

<market_watchlist_shape>
Each watchlist row should include:

- Threat/opportunity.
- Current evidence.
- Why it matters.
- Project response requirement.
- Watch cadence.
- Trigger threshold.
- Affected requirement IDs.
</market_watchlist_shape>

<citation_rules>
Final user summaries should link to web sources used. Internal Bible artifacts should store source URLs in the relevant artifact so future refreshes can verify them.
</citation_rules>
